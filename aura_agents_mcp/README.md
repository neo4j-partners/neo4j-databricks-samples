# Neo4j Aura Agent MCP on Databricks

Connect a Neo4j Aura Agent's MCP server to Databricks as an external MCP connection. The agent exposes graph query tools (Cypher templates, vector search, Text2Cypher) over the Model Context Protocol, and Databricks consumes them through its external MCP integration.

## Two OAuth Systems

An Aura Agent has two external interfaces: a REST API and an MCP server endpoint. Both require authentication, but they use completely different OAuth systems.

The REST API authenticates through Aura API credentials. You create a client ID and secret in the Aura Console, exchange them for a bearer token at `https://api.neo4j.io/oauth/token`, and include that token in your requests. This is a standard client credentials flow and it works exactly as documented.

The MCP server endpoint rejects those same credentials with a 401. It authenticates users through Auth0 (`aura-mcp.eu.auth0.com`), not through the Aura API. The OAuth endpoints, the client registration mechanism, and the token format are all different. None of this appears in the Aura documentation as a single configuration reference.

Databricks supports Dynamic Client Registration (DCR), which automates the entire OAuth setup. This sample provides a notebook that registers the MCP server in one call, plus standalone scripts for discovering OAuth endpoints and validating the flow locally.

| | Aura REST API | MCP Server |
|---|---|---|
| **Auth provider** | Neo4j API (`api.neo4j.io`) | Auth0 (`aura-mcp.eu.auth0.com`) |
| **OAuth flow** | Client credentials | Authorization code (user login) |
| **Credentials** | Aura API key | Dynamically registered client |
| **Token endpoint** | `https://api.neo4j.io/oauth/token` | `https://aura-mcp.eu.auth0.com/oauth/token` |
| **Token audience** | `https://api.neo4j.io` | `https://console.neo4j.io` (must be omitted, not set) |

## Prerequisites

- A deployed Neo4j Aura Agent with **External** access and **MCP server** enabled (see [Aura Agent Setup](#aura-agent-setup))
- [Databricks CLI](https://docs.databricks.com/dev-tools/cli/install.html) configured (`databricks auth login`)
- [uv](https://docs.astral.sh/uv/) installed (for local testing scripts)
- A browser for the OAuth test flow

## Quick Start: Databricks Registration

The fastest path to a working Databricks MCP connection uses DCR, which handles OAuth discovery and client registration automatically.

```bash
cd aura_agents_mcp

# 1. Copy .env.sample and set your MCP server URL
cp .env.sample .env
# Edit .env → set MCP_SERVER_URL

# 2. Push the MCP server URL to Databricks secrets
./setup-secrets.sh

# 3. Import register_mcp_server.ipynb into Databricks and run it
```

The notebook calls `register_mcp_server_via_dcr()`, which walks the same OAuth discovery chain described below, registers a client with Databricks' redirect URI, and creates the Unity Catalog connection. The MCP server then appears under **Agents > MCP Servers** and is available in AI Playground.

### Local Testing (optional)

Validate the OAuth flow from your local machine before touching Databricks:

```bash
# Discover OAuth endpoints and register a client
uv run discover-oauth.py

# Add the printed OAUTH_CLIENT_ID and OAUTH_CLIENT_SECRET to .env

# Test the full OAuth + MCP flow (opens browser)
uv run test-mcp-auth.py
```

## Aura Agent Setup

If you already have an Aura Agent with MCP enabled, skip to [Quick Start](#quick-start).

1. Go to [console.neo4j.io](https://console.neo4j.io) and select **Agents** in the left-hand menu
2. Click **Create with AI** (or **Create Agent** for manual configuration)
3. Select your Neo4j instance (must be running)
4. If using "Create with AI," provide an embedding provider, model, and a prompt describing the agent's purpose
5. Review the generated tools (Cypher Templates, Similarity Search, Text2Cypher)
6. Click the **...** menu on the agent → **Configure**
7. Under **Access**, select **External** and toggle **Enable MCP server** on
8. Save the agent
9. Click **...** → **Copy MCP server endpoint** — this is your `MCP_SERVER_URL`

External agents are billed at $0.35/hour while active. Switch back to Internal when not in use.

## OAuth Discovery: How the Auth Settings Were Found

The MCP specification defines an OAuth discovery flow built on three RFCs. Each step returns a URL pointing to the next, forming a chain that starts from the MCP server itself. The `discover-oauth.py` script automates this chain; the steps below explain what it does and why.

### Step 1: Probe the MCP Server

Send any request to the MCP endpoint without authentication. The 401 response includes a `www-authenticate` header with a `resource_metadata` URL, following [RFC 9728](https://datatracker.ietf.org/doc/html/rfc9728) (OAuth 2.0 Protected Resource Metadata).

```
POST https://mcp.neo4j.io/agent?project_id=...&agent_id=...
→ 401 Unauthorized
  www-authenticate: Bearer resource_metadata=https://mcp.neo4j.io/.well-known/oauth-protected-resource/...
```

### Step 2: Fetch Resource Metadata

GET the resource metadata URL. The response identifies which authorization server this resource trusts.

```json
{
  "authorization_servers": ["https://aura-mcp.eu.auth0.com"],
  "scopes_supported": ["openid", "profile", "email"],
  "resource": "https://mcp.neo4j.io/agent?project_id=...&agent_id=..."
}
```

### Step 3: Fetch Authorization Server Metadata

GET the authorization server's well-known endpoint. This follows [RFC 8414](https://datatracker.ietf.org/doc/html/rfc8414) and returns every endpoint needed for the OAuth flow.

```json
{
  "authorization_endpoint": "https://aura-mcp.eu.auth0.com/authorize",
  "token_endpoint": "https://aura-mcp.eu.auth0.com/oauth/token",
  "registration_endpoint": "https://aura-mcp.eu.auth0.com/oidc/register",
  "scopes_supported": ["openid", "profile", "email", "offline_access"],
  "code_challenge_methods_supported": ["S256", "plain"]
}
```

This is Auth0, not the Aura API. Different issuer, different token endpoint, different client credentials.

### Step 4: Register a Client

The authorization server supports [RFC 7591](https://datatracker.ietf.org/doc/html/rfc7591) dynamic client registration. POST to the registration endpoint to get a `client_id` and `client_secret` configured for your redirect URI.

```bash
curl -s https://aura-mcp.eu.auth0.com/oidc/register \
  -X POST -H "Content-Type: application/json" \
  -d '{
    "client_name": "neo4j-mcp-databricks",
    "redirect_uris": ["http://localhost:8400/callback"],
    "grant_types": ["authorization_code"],
    "response_types": ["code"],
    "token_endpoint_auth_method": "none"
  }'
```

The response contains `client_id` and `client_secret`. These are the values Databricks needs.

### The Audience Trap

If you include an `audience` parameter in the authorization request, Auth0 issues a token scoped to that audience. The MCP server validates tokens against `audience=https://console.neo4j.io`, which is Auth0's default when no audience is specified. Setting any explicit audience, even `https://console.neo4j.io` directly, can produce a token format the MCP server rejects.

The fix: omit the `audience` parameter entirely. Both scripts in this sample do this, with comments explaining why.

## Configuring Databricks

### Option A: Dynamic Client Registration (recommended)

Use the `register_mcp_server.ipynb` notebook. It calls `register_mcp_server_via_dcr()` which handles everything: OAuth discovery, client registration with Databricks' redirect URI, and Unity Catalog connection creation.

```python
from databricks.sdk import WorkspaceClient
from databricks_mcp import register_mcp_server_via_dcr

connection_url = register_mcp_server_via_dcr(
    connection_name="neo4j_aura_agent",
    mcp_url=dbutils.secrets.get("aura-mcp-secrets", "mcp-server-url"),
    workspace_client=WorkspaceClient(),
)
```

### Option B: Manual configuration (reference)

If DCR is not available, configure the connection manually in the Databricks UI. Note that the manual form does not support query parameters in the base path, so DCR is the recommended approach.

**Step 1 — Connection basics:**

| Field | Value |
|-------|-------|
| Connection name | Your choice (e.g., `neo4j_aura_agent`) |
| Connection type | HTTP |
| Auth type | OAuth User to Machine Per User |
| OAuth provider | Manual configuration |

**Step 2 — Authentication:**

| Field | Value |
|-------|-------|
| Host | `https://mcp.neo4j.io` |
| Port | `443` |
| OAuth scope | `openid profile email` |
| Client ID | From `discover-oauth.py` output |
| Client secret | From `discover-oauth.py` output |
| Authorization endpoint | `https://aura-mcp.eu.auth0.com/authorize` |

**Step 3 — Connection details:**

| Field | Value |
|-------|-------|
| Token endpoint | `https://aura-mcp.eu.auth0.com/oauth/token` |
| OAuth credential exchange method | Header and body |

**Limitation:** The base path field rejects query parameters (`?project_id=...&agent_id=...`), which Neo4j's MCP URL requires. This is why DCR is the recommended approach — it bypasses the form entirely.

## Running the Scripts

### setup-secrets.sh

Pushes `MCP_SERVER_URL` from `.env` to Databricks secrets.

```bash
./setup-secrets.sh                           # Default profile, scope: aura-mcp-secrets
./setup-secrets.sh my-profile                # Custom Databricks CLI profile
./setup-secrets.sh my-profile my-scope       # Custom profile and scope
```

### register_mcp_server.ipynb

Databricks notebook that reads the MCP URL from secrets and registers the MCP server via DCR. Import into Databricks and run.

### discover-oauth.py

Discovers OAuth endpoints from any MCP server URL and optionally registers a client. Useful for understanding the OAuth flow or configuring non-Databricks clients.

```bash
uv run discover-oauth.py                                # Discover + register client
uv run discover-oauth.py --no-register                  # Discovery only
uv run discover-oauth.py --redirect-uri https://...     # Custom redirect URI
```

### test-mcp-auth.py

Tests the full OAuth + MCP pipeline locally. Opens a browser for Auth0 login, captures the callback, exchanges for a token, and verifies the token works against the MCP endpoint.

```bash
uv run test-mcp-auth.py
```

Requires `OAUTH_CLIENT_ID` in `.env` (from `discover-oauth.py` output).

## Troubleshooting

**401 from MCP server after token exchange**
The most common cause is an `audience` parameter in the authorization URL. Verify the token's `aud` claim:

```bash
python3 -c "import json,base64; t='YOUR_TOKEN'; p=t.split('.')[1]; print(json.loads(base64.urlsafe_b64decode(p+'==')))"
```

The audience should be `['https://console.neo4j.io', 'https://aura-mcp.eu.auth0.com/userinfo']`. If it contains `api/v2/` or any other value, the authorization request included an audience parameter.

**Dynamic client registration fails**
Ensure `redirect_uris`, `grant_types`, and `response_types` are all present in the registration request.

**Callback server not receiving the code**
Check that no other process is using port 8400. Set a different port via `OAUTH_REDIRECT_URI=http://localhost:9999/callback` in `.env`.

**Databricks connection fails after configuring**
The OAuth client must be registered with Databricks' redirect URI, not localhost. Run `discover-oauth.py --redirect-uri <databricks-redirect>` to register a new client.

**"Invalid base path" error in Databricks manual configuration**
The Databricks UI rejects query parameters in the base path field. Neo4j's MCP URL requires `?project_id=...&agent_id=...`. Use DCR via the notebook instead of manual configuration.
